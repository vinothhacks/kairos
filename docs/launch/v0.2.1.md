# kairos v0.2.1 launch — install hotfix (build-in-public)

Drafts only. Not posted yet.

## Angle

The audit-story release shipped with a broken install one-liner. An external user reported it 24h after the v0.2.0 announcement. v0.2.1 fixed it in 47 minutes, retroactively for every prior release link. Same build-in-public posture as v0.2: lead with what was broken, not what's new.

## Single launch post (LinkedIn, ~1100 chars)

```
Embarrassing v0.2 update.

I shipped kairos v0.2.0 yesterday with a "45 audit findings closed" story.

I missed one.

Both install scripts have been calling `uv tool install kairos-agent`
since v0.1.0 — but I never published the package to PyPI.
Every external user hit this:

  irm .../v0.2.0/install.ps1 | iex
  -> found uv, installing as a uv tool
    × No solution found when resolving dependencies:
    because kairos-agent was not found in the package registry
  ! uv tool install failed

Caught by an external user 24h after launch. The install one-liner
was a SHIPPED feature; I never tested it from a fresh terminal.

47 minutes later, v0.2.1 is live:

→ PyPI: https://pypi.org/project/kairos-agent/0.2.1/
→ Trusted Publishing (OIDC, no secrets in repo)
→ release.yml gates publish behind a green test+lint matrix
  on the tagged SHA — the v0.2 pre-tag-CI lesson, in code

Bonus: install.ps1 has no version pin. So as soon as v0.2.1 landed
on PyPI, the original v0.1.0 / v0.1.1 / v0.2.0 install links also
started working. The fix is retroactive.

Lessons (filed in memory_outbox):
1. The install one-liner is a feature. Smoke it from a fresh shell.
2. CI gate idea: `pip install --dry-run <pkg>` on every release
   commit. Fail if the package is not resolvable on the same SHA.
3. Trusted Publishing > stored API tokens. Always.

Detection → working install on PyPI: 47 min.
Detection → external user reproed: 24h.

Test the boring stuff.

#buildinpublic #python #llm #mcp #kairos
```

## What if it flops

If the v0.2.1 post gets less reach than v0.2 (likely — it's a hotfix, not a feature drop), that's fine. The point is the build-in-public credibility, not the impressions. Don't add a carousel; don't pin it. Move on to v0.3.0 (the 30 KAI2-* findings) and let the hotfix stand on its own.

## What NOT to say

- "Surprise! There was a bug." (No suspense. Lead with the bug.)
- "Quick patch release." (Hide-the-mistake framing. Wrong.)
- "Improved the install experience." (Marketing voice. Wrong.)
- "We've published kairos to PyPI." (Forward-looking framing. Wrong — the right framing is past-broken / now-fixed.)

The audit-story angle from v0.2 doubles down here: own the gap, ship the fix, document the recurrence guard, move on.
