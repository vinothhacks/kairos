# kairos v0.2.0 LinkedIn launch kit (audit-story angle)

**Goal of this kit:** ship the v0.2.0 release with a credible, technical, build-in-public story that turns a 45-bug audit into trust instead of damage. Single launch post + 6-slide carousel + engagement plan. Zero auto-posting; every piece is a draft for human review.

**Voice:** confident, technical, specific. No em-dashes (project rule). No hype words ("revolutionary", "game-changer"). Use real numbers.

---

## A. The single launch post (LinkedIn primary)

**Length:** 1300-1500 chars. Native to LinkedIn's "see more" cutoff (~210 chars).

**Hook line (above the fold):**

```
I shipped kairos v0.1 last week. Then I audited my own product and found 45 bugs.
```

**Body:**

```
I shipped kairos v0.1 last week. Then I audited my own product and found 45 bugs.

kairos is a CLI that picks the right LLM technique (RAG / ReAct / Reflexion) for any task by querying its own wiki. Every model call goes through llm-mcp, so it works with zero API keys.

After v0.1.1 shipped, I ran the full audit playbook against my own code:
6 Critical findings.
14 High.
16 Medium.
7 Low.
2 Info.

The most embarrassing one (KAI-005): my README claimed kairos populated a wiki_index SQLite cache for fast selector lookups. The schema existed. The reads existed. The WRITE PATH NEVER FIRED. The selector was filesystem-walking on every call and pretending it had a cache.

Today I shipped v0.2.0 with all 45 findings closed AND seven features that v0.1 had documented but never actually implemented:
1. .kairos/config.toml parser
2. Real wiki_index + wiki_relations write paths
3. Plugin Runner ABC with entry_points discovery
4. kairos feedback CLI
5. MCPLLMClient retries + exponential backoff
6. Real kairos doctor probe (was hard-coded "ok")
7. Optional --llm-rerank for tie-breaks

Tests went from 48 to 88. Coverage hit 77%. ruff + mypy strict clean across src/ AND tests/. Zero breaking changes.

Why share this? Because the LLM tooling space is full of demos that don't survive contact with real use. Auditing your own code, in public, with a numbered finding list, is how you build trust faster than another launch tweet.

The audit document, the v0.2 changelog, and the migration guide are all in the repo.

github.com/vinothhacks/kairos

#LLM #Python #OpenSource #DeveloperTools
```

**Image:** screenshot of the 30-second-demo block from the README (the top-3 ranking table). Or a clean screenshot of the audit ISSUES.md with a couple of findings visible.

**Posting time (IST):** Tue or Wed, 09:30 - 10:30 IST. Avoids US morning rush; lands in EU lunch and IN/SG mid-day.

## B. The 6-slide carousel (companion piece, post 24h later)

LinkedIn carousels get ~3x reach of single-image posts when the first slide is good. Each slide is 1080x1080 (Canva or Figma).

### Slide 1 — Title

Subtitle: "Here are the 6 worst ones."
Footer: `kairos v0.2.0 release` + GitHub icon.

### Slide 2 — KAI-001 (init --force destroyed work)
```
KAI-001 — CRITICAL

`kairos init --force` overwrote
AGENTS.md without backups.

A user typo could nuke a hand-edited
schema with no recovery.

Fix: write *.bak.<UTC-timestamp>
sibling files BEFORE overwrite.
```
Footer: `Severity: Critical | Surface: filesystem`

### Slide 3 — KAI-002 (re-ingest clobbered raw bytes)
```
KAI-002 — CRITICAL

Re-ingesting a paper with the same
title silently overwrote the bytes
of the previous version in raw/.

You'd lose your immutable source.

Fix: hash-compare. Same bytes = no-op.
Different bytes = write `<name>_2.md`.
```
Footer: `Severity: Critical | Surface: ingest`

### Slide 4 — KAI-005 (wiki_index was dead)
```
KAI-005 — CRITICAL

README: "selector consults wiki_index
for fast lookups."

Reality: schema existed. Reads existed.
WRITE PATH WAS NEVER WIRED.

The selector was walking the filesystem
on every call.

Fix: WikiIndexer.upsert_page() now
fires on ingest / lint / save-to-wiki.
```
Footer: `Severity: Critical | Surface: persistence`

### Slide 5 — KAI-007 (Rich ate the wikilinks)
```
KAI-007 — HIGH

`kairos query` answers contain
[[wikilinks]] - the whole point.

Rich was parsing them as markup
and eating them in console output.

Fix: console.print(text, markup=False).

One line. Three months of mystery.
```
Footer: `Severity: High | Surface: CLI rendering`

## D. What if it flops

If the launch post gets <50 reactions in 24h:
- Don't repost. Don't beg.
- Repackage the audit as a Substack-length blog post with full findings table (the audit ISSUES.md is already in the repo).
- Wait two weeks, then post under a different angle: "I ran a 45-finding audit on a 2,500-LOC LLM tool. Here is the methodology."

If a finding gets weaponized in comments ("you shipped this BROKEN to PyPI?!"):
- Reply once, with the v0.2 fix link, and don't argue. The CHANGELOG entry is the receipt.
